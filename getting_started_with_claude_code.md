# Claude Code スタートガイド

このドキュメントは、解答用紙メーカープロジェクトを Claude Code に引き継ぐ際の**最初の一週間で行う作業手順**を示します。

---

## 0. 引き継ぎ前のチェックリスト

Claude Code でセッションを開始する前に：

- [ ] `answer_sheet_maker_v4.html`（最新の動作版）をローカルに保存
- [ ] `requirements.md`（要件定義書）をローカルに保存
- [ ] このファイル（`getting_started_with_claude_code.md`）もローカルに保存
- [ ] GitHubアカウント準備（GitHub Pages 用）
- [ ] ローカル開発環境：Node.js 推奨（Vite等を使う可能性のため）

## 1. プロジェクトのセットアップ（30分）

### 1.1 リポジトリ作成

Claude Code で以下を実行：

```bash
mkdir answer-sheet-maker
cd answer-sheet-maker
git init
```

### 1.2 ファイル配置

Claude Code に以下を依頼：

> 「`answer_sheet_maker_v4.html` を `index.html` としてプロジェクトルートに配置してください。
> 同時に `answer_sheet_maker.html` というファイル名でもコピーを残してください（配布用）。
> README.md、CHANGELOG.md、LICENSE（MIT）も作成してください。」

### 1.3 GitHub Pages のセットアップ

```bash
# GitHubにリポジトリを作成（手動）
git remote add origin https://github.com/mitsuki-0526/answer-sheet-maker.git
git add .
git commit -m "Initial commit: v0.4.0 baseline"
git push -u origin main
```

GitHub の設定で：
- Settings → Pages → Source を「main branch」に
- URL が `https://mitsuki-0526.github.io/answer-sheet-maker/` で公開される

### 1.4 自動デプロイの設定

Claude Code に以下を依頼：

> 「`.github/workflows/deploy.yml` を作成して、main ブランチへのプッシュで自動的に GitHub Pages がデプロイされるようにしてください。`requirements.md` 第9.4節を参照してください。」

## 2. 最初の機能追加：タイトル編集UI（v0.5）

### 2.1 Claude Code への指示プロンプト

```
解答用紙メーカーにタイトル編集機能を追加してください（優先度1の機能）。

【参照ドキュメント】
- docs/requirements.md 第6章「優先度1：タイトル編集UI」
- docs/requirements.md 第4章「必須DOM要素・ID」「必須クラス名」

【実装内容】
1. 用紙ヘッダーの `.sheet-header .title` 要素をクリックで直接編集可能にする
   - `contenteditable="true"` を付与
   - フォーカス時にハイライト表示
   - ぼかし時に `data.title` を更新（再描画しない、カーソル位置維持）
2. タイトルバー中央の文書名表示（`.doc-name`）と同期する
3. JSON保存時にタイトルが反映されることを確認

【テスト】
実装後、requirements.md 第5章のチェックリストをすべて手動で確認してください。
特に：
- タイトルを変更してJSON保存→読込で復元されるか
- 印刷プレビューでタイトルが反映されるか
- 既存の選択・書式編集が壊れていないか

【コミットメッセージ】
`feat: add inline title editing UI`
```

### 2.2 受け入れテスト

実装が完了したら、以下を自分でチェック：

1. タイトル「2学期期末テスト 数学」をクリック → 編集できる
2. 適当に「中間テスト 数学」に書き換えてフォーカスを外す
3. JSONとして保存
4. ページをリロード
5. 保存したJSONを読込
6. タイトルが「中間テスト 数学」になっている
7. タイトルバー中央の表示も連動する
8. 印刷プレビューで反映される
9. コンソールにエラーが出ていない

## 3. 次の機能：Undo / Redo（v0.6）

### 3.1 設計判断

Claude Code に「**Undo/Redo の実装方針**」を相談してください：

```
Undo/Redo 機能を実装したいです。requirements.md 第6章「優先度2」を読み、
以下を検討してから実装方針を提案してください：

1. スナップショット方式 vs コマンドパターン
2. contenteditable のローカル履歴との競合をどう避けるか
3. 履歴上限とメモリ使用量
4. どのタイミングで履歴を保存するか（debounce 必要？）
5. キーボードショートカット衝突の確認

実装前に提案を見せてください。
```

### 3.2 段階実装の推奨

Undo/Redo は複雑なので、以下のように段階的に：

**フェーズA：基本動作**
- 履歴スタックの実装
- Ctrl+Z / Ctrl+Y キーバインド
- 大問追加・削除のUndo/Redo

**フェーズB：拡張**
- セル内容編集のUndo/Redo（debounce）
- テキストボックス操作のUndo/Redo

**フェーズC：UI連携**
- ツールバーのUndo/Redoボタンを有効化
- 履歴がない時はボタンを無効化

各フェーズでテストを挟む。

## 4. プロジェクトの育て方（中長期）

### 4.1 イテレーションの基本サイクル

```
1. requirements.md の優先度1のタスクを選ぶ
   ↓
2. Claude Code に上記テンプレートで依頼
   ↓
3. 受け入れテストを実施
   ↓
4. 問題なければコミット & プッシュ
   ↓
5. CHANGELOG.md を更新
   ↓
6. requirements.md のチェックリストを更新（✅マーク）
   ↓
7. 次のタスクへ
```

### 4.2 1日1機能ペースの推奨

- **平日**：要件確認・テスト（30分〜1時間）
- **週末**：Claude Code でまとめて実装（2〜3時間）

無理せず、確実に動くものを積み上げる。

### 4.3 困った時のリカバリ

**コードが壊れて元に戻したい：**
```bash
git log              # コミット履歴を見る
git checkout <hash>  # 特定のコミットに戻る
```

**Claude Code に間違った指示をしてしまった：**
```bash
git diff             # 何が変わったか確認
git checkout .       # 全部元に戻す
```

**機能リグレッションが起きた：**
1. `requirements.md` 第5章のチェックリストを再確認
2. どの機能が壊れたか特定
3. Claude Code に「〇〇機能のリグレッションを修正してください」と依頼
4. 直近のコミットと比較してもらう

## 5. Claude Code とのコミュニケーション TIPS

### 5.1 効果的なプロンプトの書き方

**❌ よくない例：**
> 「タイトルを編集できるようにして」

**✅ よい例：**
> 「タイトル編集機能を追加してください。
> - 参照：requirements.md 第6章 優先度1
> - 既存の `#title` 要素を使う（変更禁止）
> - 完了後、第5章のチェックリストでリグレッション確認
> - コミットメッセージは `feat: add inline title editing`」

### 5.2 セッション開始時の決まり文句

新しい Claude Code セッションを開いた時、毎回これを最初に伝える：

```
このプロジェクトは「解答用紙メーカー」です。
詳細は docs/requirements.md を読んでください。
特に第4章（必須ID）、第5章（既存機能カタログ）は必ず確認してください。
今日は〇〇の作業をします。
```

### 5.3 動作確認は必ず自分で

Claude Code は「動くと思います」と言っても、実際にブラウザで開かないと分からないバグがあります。**ファイルの保存後、必ず自分でブラウザで開いて確認**してください。

### 5.4 デバッグ依頼

バグ報告は具体的に：

**❌ よくない例：**
> 「動かない」

**✅ よい例：**
> 「タイトル編集後、JSON保存ボタンを押しても変更前のタイトルでファイル名が付く。
> 再現手順：
> 1. タイトルを『テスト』に変更
> 2. フォーカスを外す
> 3. 保存ボタンを押す
> 4. ファイル名が古いタイトルになっている
> 開発者ツールのコンソールにエラー：〇〇」

## 7. トラブルシューティング

### 7.1 GitHub Pages が反映されない

- Settings → Pages の設定確認
- Actions タブでデプロイログ確認
- ブラウザのキャッシュをクリア（Ctrl+Shift+R）

### 7.2 印刷が変

- `@media print` ルールを再確認
- ブラウザ別の挙動差（Chrome推奨）
- A4余白設定（@page）

### 7.3 大きなJSONが読み込めない

- LocalStorage の上限は通常5MB
- 画像のBase64で圧迫している可能性
- 画像挿入機能実装時に「外部リンクモード」も用意

## 8. 重要な原則

### 8.1 単一HTMLファイル維持
v0.10 までは単一HTMLファイル構成を維持してください。配布のしやすさが最優先です。

### 8.2 リグレッション禁止
**新機能を追加した結果、既存機能が壊れることは絶対に避ける**。怪しい場合は実装をストップして相談。

### 8.3 学校現場でのテスト
実装した機能は、可能な限り**実際に学校の業務で使ってみる**。机上で完璧でも、現場で使えないことは多い。

### 8.4 同僚へのヒアリング
v1.0 リリース前に、所属校の同僚3〜5名に試してもらい、フィードバックを取る。
