## 1. プロジェクト概要

### 1.1 ツール名
**解答用紙メーカー（Answer Sheet Maker）**

### 1.2 目的
中学校・高校の教員が定期テスト・確認テスト用の**解答用紙を、Word や Excel よりも素早く・きれいに**作成できるWebアプリケーション。

### 1.3 解決する課題
- Excel での解答用紙作成における「セル幅調整地獄」
- Word での「テキストボックス位置調整の煩雑さ」
- 教科を問わず利用できる汎用ツールの不在
- テンプレート保存・再利用のしにくさ

### 1.4 ターゲットユーザー
- **第一義：** Mitsuki（作者）と所属校の同僚教員
- **拡大想定：** 中学校・高校の全国の教員（教科を問わない）

### 1.5 設計思想
- **Studyaid D.B.（数研出版）を参考にした構造化データモデル**
- **Microsoft Word 365 風のリボンUI** で学習コストゼロ
- **PDF出力はブラウザ印刷で代替**（jsPDFなどの重い依存を避ける）
- **単一HTMLファイル**で完結し、配布が容易

## 2. 現状分析（v0.4 時点）

### 2.1 ファイル構成
```
/
└── answer_sheet_maker_v4.html   (約2,340行、92KB)
    ├── <style> ... </style>     (CSS、約750行)
    ├── <body> ... </body>       (HTML、約600行)
    └── <script> ... </script>   (JavaScript、約990行)
```

### 2.2 使用技術
- **Vanilla JavaScript**（フレームワークなし、ビルドツールなし）
- **プレーンCSS**（Tailwind等は不使用、CSS変数活用）
- **Inline SVG**（Fluent UI System Icons風のアイコン定義）
- **`contenteditable`** によるリッチテキスト編集
- **`document.execCommand`** で書式変更
- **Selection API + Range API** で部分フォントサイズ変更
- **CSS columns**（`column-count`, `column-rule`）で段組み

### 2.3 配布方法
- **HTML配布**：ファイルを直接渡す（USB/メール/Drive等）
- **GitHub Pages**：`https://mitsuki-0526.github.io/answer-sheet-maker/` 等で公開予定
- 両者で**同じ単一HTMLファイル**を使用

### 2.4 動作環境
- **デスクトップブラウザ**（Chrome / Edge / Firefox / Safari）
- **画面幅 1280px 以上推奨**
- **モバイル非対応**（v0.x では対応しない、後期ロードマップ）

## 3. データモデル仕様

### 3.1 ルートデータ構造
```javascript
let data = {
  title: "2学期期末テスト 数学",     // ヘッダーに表示
  questions: [ /* 大問の配列 */ ],
  textboxes: [ /* フローティングテキストボックスの配列 */ ],
  activeQuestionId: 1               // 現在ナビで展開中の大問ID
};
```

### 3.2 questions の各要素
```javascript
{
  id: 1,                    // 一意ID（数値）
  type: "writing",          // 解答欄タイプ（下表参照）
  subCount: 5,              // 小問数
  cols: 1,                  // 列数（1〜4）
  rowHeight: 8,             // 1小問の高さ(mm)
  totalScore: 25,           // 大問の合計点
  subScores: [5,5,5,5,5],   // 小問ごとの配点（subCountと長さ揃える）
  cells: {                  // 各小問のコンテンツ
    0: {
      html: "答え",                // 入力されたHTML
      vAlign: "top|middle|bottom", // 垂直配置
      align: "left|center|right|justify" // 水平配置
    }
  }
}
```

#### 3.2.1 解答欄タイプ (`type`) の一覧

| 値 | 用途 | 見た目 |
|---|---|---|
| `writing` | 記述・計算スペース | 等間隔の罫線（6mm間隔） |
| `value` | 数値・短い語句 | 罫線なし空白 |
| `choice` | 選択（〇をつける） | 選択肢ラベル付き空白 |
| `fillin` | 穴埋め | アイウエオ自動連番 |
| `drawing` | 作図・グラフ | 5mm方眼、高さ最低30mm |
| `table` | 表・行列 | グリッドセル |
| `proof` | 証明記述 | 大きな空白枠、最低50mm |

### 3.3 textboxes の各要素
```javascript
{
  id: 1,
  x: 14, y: 60,             // 用紙左上からの位置（mm）
  w: 60, h: 14,             // サイズ（mm）
  html: "テキスト",          // 内部HTML
  borderStyle: "solid",     // solid/dashed/dotted/double/none
  borderWidth: "0.5pt",     // 0.5pt/1pt/1.5pt/2pt/3pt
  bg: "transparent",        // transparent/white/gray
  align: "left"             // left/center/right/justify
}
```

### 3.4 JSON 入出力フォーマット
保存時は上記 `data` に加えて、UI設定も含める：

```javascript
{
  ...data,
  textboxIdCounter: 5,
  settings: {
    title: "...",
    paperSize: "A4",        // A4/B5/B4/A3
    layout: "縦置き 2段",     // 縦置き 1段/2段、横置き 1段/2段/3段
    centerLine: true,        // 段組み中央線
    pageNum: true,           // ページ番号
    showSubScore: false,     // 小問配点表示
    showTotalScore: false    // 大問合計表示
  }
}
```

**重要：** JSON フォーマットは下位互換性を保つこと。新規フィールドは省略可能とし、ロード時にデフォルト値で埋めること。

## 6. 改善・追加要件ロードマップ

**優先度順に列挙。完了した機能はチェック。**

### 🥇 優先度1：タイトル編集UIの追加（v0.5）

**背景：** v0.4 で `#title` 入力欄が画面外に隠されている。タイトルを変更する手段がない。

**実装方針：**
- 用紙ヘッダー（`.sheet-header .title`）を**クリックで直接編集可能**にする
  - `contenteditable="true"` を付与
  - ぼかし時に `data.title` を更新し、`render()` を呼び直さない（再描画でカーソル消失防止）
- もしくはリボン「挿入」タブまたは「用紙設定」タブに**「タイトル」フィールドを配置**
- タイトルバー中央の「ファイル名」表示と同期

**データモデルへの影響：** なし（`data.title` は既存）

**受け入れ基準：**
- タイトルを編集できる
- 編集内容が保存（JSON）に反映される
- タイトルバーの表示も連動する
- フォーカスを外したときに変更が確定する
- リグレッションなし（既存機能が動く）

---

### 🥈 優先度2：Undo / Redo（v0.6）

**背景：** 誤操作時の復元手段がない。

**実装方針：**
- **コマンドパターン**で実装するか、**スナップショット方式**で実装するか選択
- 推奨：**スナップショット方式**（シンプル、データ量も小さい）
  - `data` 全体をディープコピーして履歴スタックに保存
  - `Ctrl+Z`、`Ctrl+Y` または `Ctrl+Shift+Z`
  - 履歴上限：50件（メモリ圧迫防止）
- 履歴を保存するタイミング：
  - 大問の追加・削除
  - セル内容の変更（debounce 500ms）
  - テキストボックスの追加・削除・移動・リサイズ
  - 用紙設定の変更
- ツールバー左にUndo/Redoボタンを設置（既存の `#i-undo`/`#i-redo` アイコン定義あり）

**データモデルへの影響：**
- `history` スタック（メモリ内のみ、JSON出力には含めない）
- セル編集中の `contenteditable` 履歴とは独立して管理

**受け入れ基準：**
- 主要操作がUndo/Redoできる
- Ctrl+Z / Ctrl+Y キーボードショートカット動作
- ツールバーボタンの有効/無効状態が反映される
- contenteditable内のローカルUndoと競合しない

### 4位：自動改ページ（複数ページ対応）（v0.8）

**背景：** 現在は1ページ前提。大問が多くなると用紙からはみ出る。

**実装方針：**
- A4 縦置きの場合、用紙の使える高さは約 264mm（297 - 余白）
- 大問の高さを逐次測定し、ページに収まらない場合は**改ページ**
- `.sheet` を複数生成（`.sheet:nth-child(2)` 以降は2ページ目以降）
- ページ番号を `1 / N` 形式で動的更新
- テキストボックスは原則1ページ目のみ（または特定ページ指定）

**技術的難所：**
- `column-count` を使った段組みと組み合わせて改ページするのが難しい
- 各段の高さを正確に測定する必要がある
- CSS の `break-inside: avoid` で大問単位の分割を防ぐ
- 段組み内の自動改ページは `display: block` + 手動測定で対応するのが現実的

**受け入れ基準：**
- 大問数 10問以上でも正しく複数ページに分かれる
- 大問が分断されない
- ページ番号が動的に更新される
- 印刷時に正しくページ送りされる

### 5位：画像挿入機能（v0.9）

**背景：** 作図問題で「座標平面の画像を背景に置き、その上に生徒が書き込む」ような構成が必要。

**実装方針：**
- リボン「挿入」タブの「画像」ボタンで実装
- ファイル選択ダイアログ → 画像をBase64エンコード
- データモデルに `images: []` 配列を追加
- フローティングテキストボックスと同じ仕組みで配置（ドラッグ・リサイズ）
- 印刷時もBase64画像がそのまま出力される
- 「外部リンク参照」モードも併設（v3 で議論したように両対応）

**データモデルへの影響：**
```javascript
data.images = [
  {
    id: 1,
    x: 50, y: 100, w: 80, h: 60,  // mm
    src: "data:image/png;base64,..." または "https://...",
    srcType: "base64" | "url"
  }
]
```

**受け入れ基準：**
- ドラッグ&ドロップまたはボタンから画像を追加できる
- 移動・リサイズができる
- JSON保存に含まれる
- 印刷時に正しく出る

**実装方針：**

- 「テンプレートとして保存」ボタン → LocalStorageに保存
- プリセット例：
  - 数学定期テスト用（記述5問 + 計算5問 + 証明1問 = 100点）
  - 英語定期テスト用（選択 + 記述 + 長文）
  - 確認小テスト用（10問短答）
- 「テンプレート一覧」モーダルから読み込み

**データモデルへの影響：**
- LocalStorageキー：`asm:templates`
- 構造：`{ id, name, data, createdAt }` の配列

**受け入れ基準：**
- テンプレートを保存・読み込みできる
- 名前を付けて整理できる
- 削除できる
- プリセットが3つ以上同梱されている

## 7. コーディング規約

### 7.1 ファイル構成方針

**v0.5〜v0.9 までは単一HTMLファイル維持**（配布のしやすさを優先）。

ただし、ファイルが**1,500行を超える `<script>` ブロック**になった時点で、以下のような分割を検討：

```
/
├── answer_sheet_maker.html         # メインファイル
├── css/
│   └── style.css                   # CSSを分離
├── js/
│   ├── data.js                     # データモデル定義
│   ├── render.js                   # シート描画
│   ├── format.js                   # 書式変更
│   ├── textbox.js                  # テキストボックス
│   ├── undo.js                     # Undo/Redo
│   └── io.js                       # JSON入出力
└── assets/
    └── icons.svg                   # SVGアイコン定義
```

**ただし：**
- 配布時は**ビルドして単一HTMLに戻す**（最終成果物はsingle-file）
- ビルドツール：Vite または カスタム concatenator
- GitHub Pages では分割版をホスト、配布版は単一HTML

**Claude Code への指示：**
> 「ファイル分割が必要と判断したら、現状の単一HTMLを破壊しないように、分割版と統合版の両方をビルドできる仕組みを作ってください。」

### 7.2 JavaScript 規約

**スタイル：**
- `function 宣言` 関数を基本とする（既存コードと合わせるため）
- ES6+ 構文OK（const/let、アロー関数、テンプレートリテラル、分割代入）
- async/await は積極的に使う
- jQueryは使わない

**状態管理：**
- グローバルな `data` オブジェクトを単一の真実の源とする
- 変更は必ず `render()` を呼び直して反映
- 局所的UI状態（選択中など）は別変数で管理

**命名：**
- 関数：`camelCase`、動詞始まり（`renderQuestionList`、`addTextbox`）
- 変数：`camelCase`
- 定数：`UPPER_SNAKE_CASE`（`LAYOUT_MAP`等）
- 既存IDの大文字小文字は維持

**コメント：**
- セクション区切り：`/* ========= セクション名 ========= */`
- 日本語コメントOK
- 「なぜ」を書く（「何を」はコードで読める）

### 7.3 CSS 規約

- **CSS変数を最大限活用**（直書きHEXは新規追加禁止）
- BEMは使わず、機能的なクラス名
- `mm` 単位を用紙関連で多用（`px` ではない）
- `@media print` で印刷時の見え方を必ず確認

### 7.4 HTML 規約

- セマンティックHTML優先（`<button>`, `<input>`, `<select>`）
- アクセシビリティ属性（`aria-label`, `title`）を可能な限り
- `tabindex` でキーボード操作可能に

### 7.5 既存IDとクラスの変更について

**絶対に変更してはいけないもの：**
- 第4章「必須DOM要素・ID」の全項目
- 第4章「必須クラス名」の全項目

**理由：** 既存のJavaScriptロジックがこれらを参照しており、リネームすると全機能が破綻する。

新規追加する要素には、自由に新しい名前を付けてよい。

## 8. テスト・検証方針

### 8.1 各機能追加後の必須チェックリスト

新機能を追加するたびに、以下を **必ず手動テスト**：

1. [ ] **既存機能の動作確認**：第5章「既存機能の完全カタログ」を上から順に試す
2. [ ] **印刷プレビュー**：Ctrl+P で実物大の見栄えチェック
3. [ ] **JSON互換性**：古いJSONファイルが読み込めるか
4. [ ] **ブラウザ互換性**：Chrome / Edge / Firefox の3種で動作確認
5. [ ] **コンソールエラー**：開発者ツールでエラーが出ていないか

### 8.2 印刷チェックの観点
- A4実寸で出力されるか
- フォントが意図通りか
- 罫線が消えていないか
- 段組み中央線が用紙最下部まで伸びているか
- 大問が途中で切れていないか
- ページ番号が正しい位置か

### 8.3 サンプルデータでのテスト
プロジェクトに同梱する `samples/` フォルダ：
- `basic.json`：4大問100点の標準的なテスト
- `large.json`：15大問の長大テスト（改ページテスト用）
- `mixed.json`：全7種類の解答欄を含むテスト

### 9.1 配布方法（両採用）

#### 9.1.1 HTML 直接配布
- **対象：** 個人利用、Mitsuki + 所属校の同僚
- **方法：** 単一HTMLファイルをGoogle Drive・USB・メール添付で配布

- **URL：** `https://mitsuki-0526.github.io/answer-sheet-maker/`（予定）
- **メリット：** リンクを共有するだけで使える、アップデート即反映
- **デメリット：** インターネット接続必要

### 9.2 GitHub リポジトリ構成（推奨）

```
answer-sheet-maker/
├── index.html                  # GitHub Pages のエントリポイント
├── answer_sheet_maker.html     # 配布用（index.html とシンボリックリンク or 同内容）

│   ├── basic.json
│   ├── large.json
│   └── mixed.json
├── docs/                       # ドキュメント
│   ├── requirements.md         # この要件定義書
│   └── architecture.md         # アーキテクチャ図
└── .github/
    └── workflows/
        └── deploy.yml          # GitHub Pages 自動デプロイ
```

### 9.3 バージョン管理
- **セマンティックバージョニング**：`v0.5.0`、`v0.5.1`、`v1.0.0`
- **タグを切る**：`git tag v0.5.0`
- **CHANGELOG** を必ず更新
- 各リリースで GitHub Release を作成し、HTMLファイルを添付

### 9.4 GitHub Pages の自動デプロイ
`.github/workflows/deploy.yml`：

```yaml
name: Deploy to GitHub Pages
on:
  push:
    branches: [main]
jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Deploy
        uses: peaceiris/actions-gh-pages@v3
        with:
          github_token: ${{ secrets.GITHUB_TOKEN }}
          publish_dir: .
```

- 既存クラス名のリネーム

【完了条件】
- 新機能が動作する
- 既存機能にリグレッションがない
- コンソールエラーなし
- 印刷プレビューでも問題ない
